Semantics

Preparation

The CV page was probably the buggiest part to develop, mostly because I couldn't figure out flexbox. Didn't even manage to fit in an actual CV. To be honest, the idea of taking the filler content seriously went out the window about 20 seconds after the Curriculum Vitae/Cruiser Voler thing clicked in my head. Both are valid acronyms for CV, just in different context. One is your resume the other is short for Aicraft Carrier.

Willie D

Deciding that the filler text itself probably isn't that important, I deccided it would be fun to tell the legend of Willie D. This page really is dedicated to DD-579 USS William D Porter's story, and debunking a few myths about her in the form of a Curriculum Vitae while playing satire in the references (to the legend itself, of which the third is not confirmed true).

Sidebar

The sidebar starts off with Header level 2s because the top nav bar has the title as H1. Within this, the skill and language "progress bars" have H3s for the titles, while the UL for hobbies above is just a simple Unordered List. In the references part its a nested quote with text in a paragraph each under H3 headings to maintain the hierarchy.

Making progress

The progress bars are coded in HTML/CSS by creating boxes and setting backgrounds for them (And a bit of rounding for the edges so they don't look as...hard). This allows us to change settings by simply copy/pastaing code and changing fill width in CSS instead of making new images in Photoshop or something.

"Incoming fire has the right of way!"

The main content is very basic and straight forwards. The unordered list is formatted through CSS to not have any bullet points as it looks cleaner this way. This is a concious choice to maintain the look of the page, while also making sure the tag is clearly indicated as a list to a screenreader.

The paragraphs themselves are formatted through a separate CSS document containing some duplications to the "main" CSS file, but considering how buggy this entire ordeal was, we decided to not risk messing with something that already works just fine. If it ain't broke, don't fix it!

"Anything you do can get you shot, including nothing."

The reason we concluded that the sidebar needed to use headers in the sidebar for skills and languages over normal paragraph or quotes is that they are "headers" for the "progress bars" that themselves describe the capabilities of our little warship. Having them as coded progress bars over images means that its easier to alter, duplicate or replace them compared to having them as images. Using code is also "lighter" on the loading rather than having to use images which then need to be loaded using bandwidth.

Using quotes for the references as a little bit of trivia. All three have historical relevance, though only the third one could possibly be a real one stemming from the urban legend of USS William D Porter. This is mostly to show the use of the quote HTML element.

"The problem with taking the easy way out is that the enemy has already mined it ."

The use of paragraphs in the main CV filler is the best option because the text there is structured as paragraphs, with correct hierarchy of headers. Each subsection has the same level of importance in the setup, thus the choice of H2 headers for the sections "about me", "education" and "work experience"

The choice of an unordered list for the education is because its written as bullet points. It is neither written with the intention of flow or as a "readable paragraph". This an unordered list makes sense semantically, even if the choice was made to not display bullet points for this section or indent the text in any way.

Portfolio

For the portfolio page, I wanted to create a simple page for showcasing our projects. I figured out that it would be best to use the <figure> tag that the images of the projects are specified as self-contained content. This makes the content more semantically organized.

All images have “title” and “alt” attributes included to provide extra information about the image, and the alternative text if the image could not be displayed or viewed due to accessibility reasons. All figures have a <figcaption> tag to give the image short a caption.

All our figure elements are nested inside <section> tags so each project splits into different sections. All our section elements are again nested inside the <main> tag because the projects are the main content.

I chose to use these different tags and attributes because they provide additional information about the code. This makes the code more understandable and also instructs browsers on how to treat this information.
